## 问题2：文本块数量超过限制

### 遇到的问题
- 错误信息：`创建向量库失败: Error code: 400 - {'error': {'code': 'InvalidParameter', 'message': 'The parameter `input` specified in the request are not valid: array too long, expected an array with maximum length 256, but got an array with length 291 instead.'}}`

### 分析
- 火山方舟的嵌入模型对单次请求的文本块数量有限制，最多传256个文本块
- 课程资料分割后生成了291个文本块，超出了限制

### 解决方法
- 实现批量处理文本块的逻辑，每次处理不超过256个
- 对于第一批文本块，创建新的FAISS向量库
- 对于后续批次，创建临时向量库并合并到主向量库

### 主要改动
- 修改`create_vectorstore`函数，添加批量处理逻辑
- 每次处理256个文本块，循环直到处理完所有文本
- 添加处理进度输出

## 问题3：输入参数格式错误

### 遇到的问题
- 错误信息：`创建向量库失败: Error code: 400 - {'error': {'code': 'InvalidParameter', 'message': 'The parameter `input[0]` specified in the request are not valid: expected a string, but got `[86461 68464 ...]` instead.'}}`

### 分析
- 火山方舟的嵌入模型要求input参数是字符串类型，但传入的是数字数组
- 文本分割后的数据格式不正确，可能被错误处理成了字符编码数组

### 解决方法
- 添加代码来确保texts是Document对象列表
- 在不是Document对象时进行转换
- 跳过非字符串元素

### 主要改动
- 在`create_vectorstore`函数中，添加对texts类型的检查
- 导入`Document`类
- 实现从字符串到Document对象的转换逻辑

- 百炼平台提供了更兼容OpenAI API的接口

- 修改create_vectorstore函数，使用百炼平台的嵌入模型

## 问题7：批次大小限制问题

### 遇到的问题
- 错误信息：`调用Embedding API失败: 400 Client Error: Bad Request for url: https://dashscope.aliyuncs.com/compatible-mode/v1/embeddings`
- 响应内容：`{"error":{"message":"<400> InternalError.Algo.InvalidParameter: Value error, batch size is invalid, it should not be larger than 25.: input.contents"}}`

### 分析
- 参考文档：https://bailian.console.aliyun.com/cn-beijing/?spm=5176.29619931.J_SEsSjsNv72yRuRFS2VknO.2.2a5610d7c1Gxp0&tab=api#/api/?type=model&url=2846066
- 阿里云百炼text-embedding-v2模型的批次上限不是256，而是25
- 之前使用的256是火山方舟的限制，不适用阿里云百炼
- 批次大小设置错误导致API调用失败

### 解决方法
- 将batch_size从256改为25，符合阿里云百炼的API限制
- 添加错误处理逻辑，避免在API调用失败时出现'NoneType' object is not subscriptable错误

### 主要改动
- 修改create_vectorstore函数，将batch_size设置为25
- 在CustomEmbeddings类的embed_documents方法中添加错误处理
- 确保在API调用失败时返回空列表，避免后续操作出错

## 问题8：向量库加载失败问题

### 遇到的问题
- 错误信息：`初始化向量库失败: The de-serialization relies loading a pickle file. Pickle files can be modified to deliver a malicious payload that results in execution of arbitrary code on your machine.You will need to set `allow_dangerous_deserialization` to `True` to enable deserialization.`

### 分析
- FAISS向量库存储使用了pickle文件格式
- 为了安全，LangChain默认禁用了pickle文件的反序列化
- 需要显式设置allow_dangerous_deserialization=True来加载向量库

### 解决方法
- 在FAISS.load_local方法中添加allow_dangerous_deserialization=True参数
- 确保只从可信的来源加载向量库文件

### 主要改动
- 修改init_vectorstore函数，在FAISS.load_local调用中添加allow_dangerous_deserialization=True参数

- 完全控制请求格式，确保符合阿里云百炼的要求
- 在`init_vectorstore`函数中使用自定义类替代`OpenAIEmbeddings`
- 在`create_vectorstore`函数中也使用自定义类

### 主要改动
- 在文件顶部添加`from langchain_core.embeddings import Embeddings`和`import requests`导入
- 创建全局的`BailianEmbeddings`类，继承自`Embeddings`
- 实现`embed_documents`和`embed_query`方法，直接调用阿里云百炼API
- 修改`init_vectorstore`函数，使用`BailianEmbeddings()`替代`OpenAIEmbeddings`
- 简化`create_vectorstore`函数，移除内部的`get_embeddings`函数和`CustomEmbeddings`类定义，直接使用全局的`BailianEmbeddings`类

## 问题10：向量库不兼容导致查询无结果

### 遇到的问题
- API返回`{"answer":"在提供的课程资料中找不到相关信息。","sources":[]}`
- 明明提供了相关的PDF文件，但系统无法检索到相关信息
- `/init`端点只加载现有向量库，不会重新创建

### 分析
- 旧的向量库是用之前的嵌入模型（可能是OpenAI格式）创建的
- 新的`BailianEmbeddings`类与旧的向量库不兼容
- 向量维度或编码方式不同，导致检索失败
- 需要强制重建向量库

### 解决方法
- 为`init_vectorstore`函数添加`force_rebuild`参数
- 当`force_rebuild=True`时，删除旧的向量库文件夹，重新创建
- 修改`/init`端点，支持`force_rebuild`查询参数

### 主要改动
- 修改`init_vectorstore`函数签名：`def init_vectorstore(force_rebuild=False)`
- 添加删除旧向量库的逻辑：`shutil.rmtree("course_knowledge_base")`
- 修改`/init`端点：`async def init_knowledge_base(force_rebuild: bool = False)`
- 使用方式：`curl -X POST "http://127.0.0.1:8001/init?force_rebuild=true"`

## 问题11：查询显示“在提供的课程资料中找不到相关信息”

### 遇到的问题
- API返回`{"answer":"在提供的课程资料中找不到相关信息。","sources":[]}`
- 明明提供了相关的PDF文件，但系统无法检索到相关信息

### 分析
- 向量库已成功重建（包含 “改进 NSGA-II 算法” 的文本块），但检索环节未匹配到该文本
- 问题出在「检索参数配置」和「RAG 问答链的提示词 / 检索逻辑」，而非向量库本身。
- 修改检索器配置（放宽匹配阈值 + 扩大检索范围）
- 修改检索逻辑（使用自定义提示词）


### 解决方法
- search_kwargs={"k": 3}可以改为search_kwargs={
        "k": 15,  # 最多返回15个相似文本（默认4，扩大范围）
        "score_threshold": 0.1  # 相似度阈值设为0.1（越低匹配越宽松，0为完全不匹配）
    }
- 修改提示词：如果参考资料中包含与问题相关的内容，哪怕只有部分相关，也要详细回答；只有参考资料完全无相关内容时，才能直接回答“在提供的课程资料中找不到相关信息”

### 主要改动

- 修改提示词

## 问题14：容器启动后，显示服务不可用

### 分析
- 系统代理配置 是导致访问失败的根本原因
- Windows 会优先使用残留的代理配置转发 localhost/127.0.0.1 请求，导致访问 3001 端口时被代理拦截，返回「连接重置」。

### 解决方法
- 以管理员身份运行 PowerShell，执行命令清除代理
- 重启 Open WebUI 容器

### 主要改动
- # 清除用户级代理
  Set-ItemProperty -Path "HKCU:\Software\Microsoft\Windows\CurrentVersion\Internet Settings" -Name ProxyServer -Value ""
  Set-ItemProperty -Path "HKCU:\Software\Microsoft\Windows\CurrentVersion\Internet Settings" -Name ProxyEnable -Value 0

## 问题16：Open WebUI连接后端API出现网络错误

### 遇到的问题
- Open WebUI配置连接后端API时显示网络错误
- 我的Open WebUI在conda环境运行，项目在全局环境运行（不影响）

### 分析
- 后端API已经配置了CORS（`allow_origins=["*"]`），但API格式不兼容
- Open WebUI期望的是OpenAI兼容API格式（`/v1/models`和`/v1/chat/completions`端点）
- 原有后端只有自定义的`/ask`端点，导致验证连接失败
- 关于环境隔离：conda环境和全局环境互不影响，因为通过网络端口通信

### 解决方法
- 添加OpenAI兼容的API端点
- 添加`GET /v1/models`端点返回可用模型列表
- 添加`POST /v1/chat/completions`端点处理OpenAI格式的聊天请求

### 主要改动
- 在main.py中添加OpenAI兼容的请求/响应模型（OpenAIModel, OpenAIModelsResponse, OpenAIMessage, OpenAIChatRequest等）
- 添加`GET /v1/models`端点，返回`course-assistant`和`qwen-plus-2025-07-28`两个模型
- 添加`POST /v1/chat/completions`端点，处理OpenAI格式的聊天请求
- 该端点会从messages中提取用户问题，调用RAG链，返回OpenAI格式的响应
- Open WebUI配置时API地址需要设为`http://localhost:8001/v1`（注意加/v1）

## 问题17：评估脚本中 VectorStoreRetriever 方法不存在

### 分析
- LangChain新版本中，`VectorStoreRetriever`的API发生了变化
- 旧方法`get_relevant_documents()`已被弃用，改为`invoke()`方法

### 解决方法
- 将`retriever.get_relevant_documents(query)`改为`retriever.invoke(query)`
- LangChain的新API使用LCEL（LangChain Expression Language）风格，统一使用`invoke()`方法

### 主要改动
- 在evaluate.py的evaluate_recall_at_k()函数中，修改检索方法调用
- 保持其他逻辑不变，只更新方法调用方式
